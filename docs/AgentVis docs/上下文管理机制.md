# AgentVis MB 与 SA 上下文管理机制

> **文档范围**：Master Brain（MB）决策脑与 Sub-Agent（SA）执行单元之间的上下文管理全链路  
> **代码基础**：`src/services/planning/`

---

## 整体架构概览

AgentVis 采用 **MB + SA 分层架构**：MB 负责战略决策，SA 负责原子执行。两者分别维护独立的上下文管道，通过若干共享机制（Task Artifact Store、hitlStore、AgentSession）完成协作。

```
用户请求
    │
    ▼
┌───────────────── AgentService（FSM Owner）──────────────────┐
│                                                             │
│  ┌──────────────────────┐   ┌───────────────────────────┐  │
│  │  Master Brain (MB)   │   │  Sub-Agent (SA)           │  │
│  │  策略决策，上帝视角   │   │  原子执行，工具调用        │  │
│  │                      │   │                           │  │
│  │ MasterBrainInput     │   │ System Prompt             │  │
│  │  ┣ userIntent        │   │  ┣ BASE_TEMPLATE          │  │
│  │  ┣ memory            │   │  ┣ behaviorHint           │  │
│  │  ┣ ragEvidence       │   │  ┣ LOOP_GUIDANCE          │  │
│  │  ┣ conversationHist  │   │  ┣ inputProtocol          │  │
│  │  ┣ toolCatalog       │   │  ┣ taskExperiences        │  │
│  │  ┣ externalSkills    │   │  ┣ sandboxRuntime         │  │
│  │  ┣ skillCatalogs     │   │  ┣ toolSection            │  │
│  │  ┣ artifactIndex     │   │  ┣ externalGuides         │  │
│  │  ┣ workdirSnapshot   │   │  ┣ externalScripts        │  │
│  │  ┗ mbDecisionHistory │   │  ┣ venvConstraints        │  │
│  │                      │   │  ┣ platformInfo           │  │
│  └──────────────────────┘   │  ┗ TOOL_CALL_SELF_CHECK   │  │
│                             └───────────────────────────┘  │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │         Task Artifact Store（跨 SA 共享通道）        │  │
│   └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

## 一、MB 端上下文管理

### 1.1 MasterBrainInputBuilder — 数据聚合器

**文件**：[MasterBrainInputBuilder.ts](../../src/services/planning/agent-loop/builders/MasterBrainInputBuilder.ts)

每次 FSM 进入 `MASTER_DECISION` 状态时，`MasterBrainInputBuilder.build()` 从多个数据源聚合构建 `MasterBrainInput` 合约对象，共包含以下信息层：

| 信息层 | 来源 | 作用 |
|--------|------|------|
| `userIntent` | `AgentSession.getMessages()` 最后一条 user 消息 | 当前用户意图，含原始发送时间（sentAt）让 MB 推算耗时 |
| `memory` | `MemoryService.getMemorySnapshot(agentId, query)` | 长期事实记忆 + 摘要记忆（语义召回） |
| `ragEvidence` | `RagService.getRAGEvidence(query)` | 知识库检索结果（RAG） |
| `conversationHistory` | `AgentSession.getMessages()` 最近 N 轮 user/assistant | 短期对话上下文补充（水位线间未摘要的最新轮次） |
| `toolCatalog` | `AgentLoop.getToolCatalogEntries()` | 可用工具决策级摘要（非完整 SKILL.md） |
| `externalGuideSkills` | `SkillRetriever.retrieve(query, topK=3)` | 按用户意图语义检索的外部 Guide 技能（Top-K） |
| `externalScriptSkills` | `SkillRetriever` / 注册表精确匹配 | 用户或 MB 明确提到的 Script 技能，传递到 DISPATCH 阶段 |
| `installedSkillCatalog` | 静态全量 Guide 技能（name + description） | 确保 MB 知晓所有已安装 Guide 技能，防止语义检索遗漏 |
| `installedScriptSkillCatalog` | 静态全量 Script 技能目录 | 确保 MB 知晓可通过 `external_skill_execute` 调用的 Script 技能 |
| `taskArtifactIndex` | `TaskArtifactStore.getIndex()` | 前序 SA 中间成果的轻量索引（工具名 + 来源参数） |
| `taskArtifactObservations` | `SharedState.saObservationsSummaries` | SA 向 MB 汇报的 observations 摘要时间线 |
| `workdirSnapshot` | `SubAgentDispatcher.scanWorkdirFiles()` 聚合统计 | WORKDIR 文件系统状态（总文件数/扩展名分布/最近5个文件） |
| `projectPath` / `deliverableWorkdir` | `AgentLoopFSMIntegration` 配置 | 外部项目 cwd 切换与原交付物目录提示 |
| `sandboxMode` | Agent 安全模式配置 | 注入 MB 沙箱感知，区分 LocalAudit / ControlledNetwork / OfflineIsolated |
| `mbDecisionLog` | `SharedState.mbDecisionLog` | 最近 N 轮 MB 决策推理链，防止长任务中重复派遣或决策漂移 |
| `lastMBDecision` | `SharedState.lastMBRationale + lastMBTask` | 单轮 fallback；当 `mbDecisionLog` 不存在时保持向后兼容 |
| `hasExecutedSA` | 当前 Session 的 `sub_agent_*` tool 消息 | Round 2+ 时让 `[USER_INTENT]` 脱敏渲染，避免把原始用户消息当新请求 |
| `mbBudgetRemaining` | `LoopGovernor.getSnapshot()` | MB 预算临近耗尽时在 messages 尾部注入收尾提醒 |

### 1.2 MB Prompt 组装结构

**文件**：[MasterBrainPrompt.ts](../../src/services/planning/brain/MasterBrainPrompt.ts)

MB 的 Prompt 由头部硬约束、固定上下文、预算管理区块和尾部格式锚点组成：

```
┌─── P0（Prime Directive，最高优先级）─────────────────────────┐
│  • 角色定义（Master Brain 决策脑）                            │
│  • 三种决策类型定义（SPAWN_SUB_AGENT/REQUEST_MORE_INPUT/     │
│    RESPOND_TO_USER）                                         │
│  • 输出格式要求（JSON Schema）                                │
└──────────────────────────────────────────────────────────────┘

┌─── 固定上下文（纳入固定 token 开销，不参与渐进截断）────────────┐
│  • Identity Awareness / Character Grounding                   │
│  • AgentRules（MB 专属规则）                                  │
│  • CURRENT_TIME / USER_INTENT / WORKDIR / PROJECT_CONTEXT      │
│  • WORKDIR_SNAPSHOT（buildWorkdirSnapshot 聚合）               │
│  • MB_SANDBOX_AWARENESS（ControlledNetwork / OfflineIsolated） │
│  • MB_DECISION_HISTORY（优先）或 LAST_MB_DECISION（fallback）  │
│  • 已安装 Guide / Script 技能目录、命中的外部 Guide 技能       │
│  • OUTPUT_FORMAT_FOOTER（尾部 JSON 格式锚点）                 │
└──────────────────────────────────────────────────────────────┘

┌─── 可变区（可按预算截断）────────────────────────────────────┐
│  • TOOL_CATALOG（渐进截断，4级策略）                          │
│  • CONVERSATION_HISTORY（最近 N 轮对话）                     │
│  • MEMORY（长期事实、摘要、task_experience）                  │
│  • RAG_EVIDENCE（按 relevance 截断）                          │
│  • TASK_ARTIFACTS（前序 SA 中间成果索引）                     │
│  • 前序 SA 推理过程摘要（time-line 形式）                     │
└──────────────────────────────────────────────────────────────┘
```

### 1.3 工具目录渐进截断（4级策略）

MB Prompt 中的 `TOOL_CATALOG` 随预算压力动态截断，`BASE_TOOLS`（read/local_search/web_search/exec/file_write）已从 TOOL_CATALOG 中过滤，MB 仅见特殊工具（cron/generate_image 等）：

| Level | 移除内容 | 保留内容 |
|-------|---------|---------|
| 1 | — | 全部完整保留 |
| 2 | `whenToUse` | `whenNotToUse` + `decisionHint` + 完整 description |
| 3 | `whenToUse` + `decisionHint` | `whenNotToUse` + 完整 description |
| 4 | 所有扩展字段 | 仅 name + 首句 description + riskLevel |

### 1.4 战略连续性注入（[MB_DECISION_HISTORY] / [LAST_MB_DECISION]）

**问题**：SA 执行完成后，其最终报告在父级 `AgentSession` 中以 `role: tool`、`toolName=sub_agent_*` 保存；进入下一轮 MB 纯文本调用前，`AgentLoop` 会把 `tool` 消息转换为 `role: user` 消息以适配 LLM API。如果缺少语义围栏，MB 可能将 SA 报告误判为用户发起了新需求，导致重复派遣。

**两层防护**：

```
第一层：SA 报告语义围栏（AgentLoop.ts）
  ┌────────────────────────────────────────────────────────┐
  │ [SYSTEM: 以下是 Sub-Agent (工具名) 的执行完成报告，    │
  │  这不是用户消息]                                       │
  │ ...报告内容...                                         │
  │ [END_SA_REPORT]                                        │
  └────────────────────────────────────────────────────────┘

第二层：[MB_DECISION_HISTORY] 优先注入（MasterBrainInputBuilder）
  ┌────────────────────────────────────────────────────────┐
  │ [CONVERSATION_HISTORY]  ...用户对话历史...             │
  │                                                        │
  │ [MB_DECISION_HISTORY]   ← 最近 N 轮 MB 决策推理链      │
  │   Round 1 - Completed / SA failed                      │
  │   Decision rationale: ...                              │
  │   Dispatched task: ...                                 │
  │                                                        │
  │ [TASK_ARTIFACTS]        ...SA 执行成果...              │
  └────────────────────────────────────────────────────────┘
```

`mbDecisionLog` / `lastMBDecision` 数据流：
- **同轮写入**：`StateHandlers.handleDispatch()` 在 SA dispatch 完成后写入一条 `MbDecisionLogEntry`（含 round、rationale、task、completed/failed）
- **滑动窗口**：`PLANNING_CONSTANTS.MB_DECISION_LOG_MAX_ROUNDS = 5`，只保留最近 5 轮
- **同轮注入**：`AgentLoopFSMIntegration.syncSharedState()` → `setMbDecisionLog()` / `setLastMBDecision()` → 下一轮 MB 输入
- **Prompt 选择**：`MasterBrainPrompt` 优先渲染 `[MB_DECISION_HISTORY]`；若历史日志为空，再 fallback 到 `[LAST_MB_DECISION]`
- **跨请求持久化**：异常终止时由 `injectRationaleBeforeReturn()` 追加到 `persistContent`，JSON 降级时由 `StateHandlers` 追加 taskBlock

> [!NOTE]
> `[MB_DECISION_HISTORY]` / `[LAST_MB_DECISION]` 均不参与可变区截断，因为它们是保持 MB 战略连续性的核心上下文。

## 二、SA 端上下文管理

### 2.1 SubAgentPromptBuilder — System Prompt 组装

**文件**：[SubAgentPromptBuilder.ts](../../src/services/planning/sub-agents/SubAgentPromptBuilder.ts)

SA 的 System Prompt 按固定顺序拼接多个 Section，各 Section 以 `\n\n---\n\n` 分隔。随着沙箱、外部 Script Skill 和尾部安全锚点机制加入，当前已不再是固定 11 段：

| 编号 | Section | 内容 | 触发条件 |
|------|---------|------|---------|
| 1 | `BASE_TEMPLATE` | 核心约束：职责边界、执行红线、输出规范 | 始终注入 |
| 2 | `getBehaviorTemplate()` | 行为修饰符（careful/direct 两种风格） | `spec.behaviorHint` 有值时 |
| 3 | `LOOP_EXECUTION_GUIDANCE` | 循环执行模式说明、执行经验报告格式 | `spec.loopConfig` 有值时 |
| 4 | `formatAgentRules()` | 用户自定义角色规则（PM 角色等） | `context.data.agentRules` 有值时 |
| 5 | `buildCurrentTimePrompt()` | 当前时间（让 SA 感知时间维度） | 始终注入 |
| 6 | `buildSandboxRuntimeContextSection()` | 隔离运行时/网络沙箱约束 | `sandboxMode` 为 `ControlledNetwork` / `OfflineIsolated` 时 |
| 7 | `buildInputProtocol()` | 任务角色、背景上下文、前序 SA 报告、Artifact 快照、HITL 约束、任务上下文 JSON | 始终注入 |
| 8 | `buildTaskExperienceSection()` | 记忆系统中已有的 task_experience，含去重提示 | 有历史经验时 |
| 9 | `buildToolSection()` | 授权工具的 SKILL.md 原文；已被 Guide 命中的技能避免重复注入全文 | 始终注入 |
| 10 | `buildExternalGuideSection()` | SkillRetriever 命中的 Guide 技能全文 + 元数据；尾部列出其他已安装技能名 | 有命中技能或已安装技能目录时 |
| 11 | `buildExternalScriptSection()` | 精确命中的 Script Skill 执行元数据 | 有命中 Script 技能时 |
| 12 | `buildVenvConstraintSection()` | Python venv 路径约束（禁止装包/新建环境） | `venvPythonPath` 有值时 |
| 13 | `buildPlatformInfoSection()` | Windows 平台命令约束（dir/type 等） | Windows 环境时 |
| 14 | `TOOL_CALL_SELF_CHECK` | CoT 工具调用自检指导（建立思维框架） | `spec.loopConfig` 有值时 |

### 2.2 buildInputProtocol — 核心上下文注入

`buildInputProtocol()` 是 SA 上下文的核心组装方法，按以下顺序注入、组合：

```
## Input Protocol

> [!CAUTION] ← HITL 强制约束（最高优先级，来自 context.data.hitlOverride）
> ### 🚨 User Intervention Constraint — Must Be Followed
> ...

### Task Role
{spec.role}

### Background Context
{spec.contextSummary}  ← 可选，由 MB 提供

### Previous Sub-Agent Reports (continue from these; do not repeat completed work)
{recentToolResults: 前序 SA 的 TASK_COMPLETE 级汇报}  ← role=tool 消息 slice(-3)

### Previous Task Artifacts (do not repeat completed work)
{artifactSnapshot: Artifact Store 的完整内容}  ← 总上下文窗口 × 15%，预算不足时优先保留最新条目

### Termination Condition
{spec.terminationCondition}  ← 可选

### Task Context
{sanitizedContext JSON}  ← 已过滤敏感字段和已注入字段
```

> SA 需要历史对话时，经 MB 设置 `nextStep.includeHistory=true` ，由 `SubAgentDispatcher.buildRunnerHistoryMessages()` 将最近 N 轮 user/assistant 历史作为 Runner `messages[]` 前缀注入到首条任务指令之前，避免同一段历史同时散落在 system prompt 和 messages 里。

### 2.3 上下文隔离（sanitizeContext）

`sanitizeContext()` 在序列化 `context.data` 之前执行两类过滤：

**敏感字段过滤**（安全）：`userId`、`apiKey`、`token`、`password`、`secret`、`globalGoal`

**已注入字段排除**（防止二次序列化）：
| 原始字段 | 已注入位置 |
|---------|----------|
| `artifactSnapshot` | `buildInputProtocol()` Artifact Section |
| `recentToolResults` | `buildInputProtocol()` 前序 SA 报告 Section |
| `agentRules` | Section 4（formatAgentRules） |
| `taskExperiences` | 历史经验区块（buildTaskExperienceSection） |
| `hitlOverride` | `buildInputProtocol()` 顶部 CAUTION 块 |

### 2.4 每步 LLM 调用 — 尾部安全锚点

**文件**：[SubAgentLLMCaller.ts](../../src/services/planning/agent-loop/callers/SubAgentLLMCaller.ts)

SA 每步 LLM 调用（`callWithContext`）的消息组织：

```
[system]  ← 上述 System Prompt

[user]    ← 可选历史对话前缀（includeHistory=true 时）
[user]    ← 初始任务指令（首步）
[assistant] ← LLM 响应
[tool]    ← 工具执行结果
[user]    ← 可选 additionalInstructions（策略调整 / 预算警告 / context reset 指令）
           + "---" + 可选 SAFETY_FOOTER_TEXT（仅 subAgentSafetyFooterEnabled=true）
           + "---" + 可选 persistedIntervention（HITL 用户介入，始终放在最末尾）
```

> **关键设计**：尾部 user 消息是 SA 每步调用的注意力热区。`SAFETY_FOOTER_TEXT` 在开关启用时放入该热区；如果用户通过 HITL 介入，`persistedIntervention` 会追加在 Safety Footer 之后，优先级更高，并在后续每步持续出现。

> **MaxTokens Budget**：单次调用的服务商输出 Token 额度（budget）与此类输入上下文的组织形式相互独立。常规的子智能体（SA）调用会采用 subAgent 输出配置规范（基准为 32K，并在发生显式参数拒识时提供一个降级至 24K 的备用方案）。下文提及的上下文窗口比例仅用于衡量累积的输入历史数据，绝不能被用作模型输出的上限界定。如果被接纳的响应因耗尽了输出额度而被迫终止，系统会在执行工具之前将其直接丢弃，并使用拆分任务的指令重新发起重试。

## 三、SA 三级递进上下文压缩策略

当 SA 执行长链路任务时，`messages[]` 会随工具调用结果累积而膨胀。Runner 通过三级压缩机制维持上下文健康：

**文件**：[SubAgentRunner.ts](../../src/services/planning/sub-agents/SubAgentRunner.ts)

```
               token 占比（总上下文窗口）
  0%─────────────45%──────────────────85%────100%
               │                    │
               ▼ L2 触发            ▼ L1 触发
  正常执行      上下文重置            梯度压缩（高位安全兜底）
               （主动清理）           （被动激进截断）
```

### 3.1 L1 梯度压缩（触发阈值：85%总窗口）

**高位安全兜底**，仅在 L2 未能充分释放时触发。

**策略**：`compressHistoricalToolOutputs()`

历史区（保护区以外）的 tool 消息按两种压力模式处理：

| 条件 | 处理方式 |
|------|---------|
| **无压力**（token < 85%窗口） | 标准梯度压缩：< 5K tokens 的完整保留 |
| **有压力 + 有 assistant 分析** | 激进压缩为元信息 + 引用提示（已有 LLM 分析结论） |
| **有压力 + 无 assistant 分析** | 标准梯度压缩（保留更多原始内容） |

工具特定上限：`SUB_AGENT_COMPRESS_THRESHOLD_L1 = 8000 tokens`

### 3.2 L2 上下文重置（触发阈值：45%总窗口 且剩余步数≥3）

**主动清理机制**，给 SA 机会输出结构化摘要后重新开始：

```
Step N: token > 45% 总窗口
    ↓ Runner 注入 CONTEXT_RESET_INSTRUCTION（via additionalInstructions）

Step N+1: SA 输出 ---CONTEXT_SUMMARY--- 标记的结构化摘要
    ↓ Runner 拦截摘要，在终止信号检测之前（continue 跳过检测）
    ↓ 清空 messages[]（保留 system prompt）
    ↓ 注入摘要为新 user message

Step N+2: SA 从摘要继续执行未完成任务（全新上下文）
```

**关键设计**：
- 支持**无限次**重置（`contextResetCount` 计数，每次重置后解锁允许再次触发）
- `toolCallSteps` 和 `totalToolCalls` **不归零**，保持总预算控制
- SA 未在 2 步内响应时，`buildMechanicalSummary()` 机械兜底

### 3.3 L3 预算警告（步数占比触发）

在步数即将耗尽时注入收尾提示，引导 SA 主动完成：

| 触发条件 | 注入内容 | 常量 |
|---------|---------|------|
| `toolCallSteps / maxSteps > 85%` | 轻量收尾提示 | `SUB_AGENT_BUDGET_WARNING_RATIO = 0.85` |
| `toolCallSteps / maxSteps > 95%` | 最终强警告 | `SUB_AGENT_BUDGET_CRITICAL_RATIO = 0.95` |

### 3.4 关键常量一览

**文件**：[PlanningConstants.ts](../../src/services/planning/PlanningConstants.ts)

```typescript
SUB_AGENT_TOKEN_PRESSURE_RATIO      = 0.85  // L1 梯度压缩触发
SUB_AGENT_CONTEXT_RESET_RATIO       = 0.45  // L2 上下文重置触发
SUB_AGENT_CONTEXT_RESET_MIN_REMAINING_STEPS = 3  // L2 最小剩余步数
SUB_AGENT_BUDGET_WARNING_RATIO      = 0.85  // L3 首次预算警告
SUB_AGENT_BUDGET_CRITICAL_RATIO     = 0.95  // L3 最终预算警告
SUB_AGENT_BUDGET_CHECKPOINT_REMAINING_STEPS = 5   // 临近耗尽时触发预算 Checkpoint
SUB_AGENT_BUDGET_EXTENSION_MAX_ITERATIONS = 20    // 单次预算扩展最大步数
SUB_AGENT_BUDGET_EXTENSION_MAX_COUNT = 2          // 单次 SA 最多扩展次数
TOOL_CALLS_HARD_LIMIT               = 200   // 工具调用总数安全阀

```

## 四、Task Artifact Store — 跨 SA 成果持久化

**文件**：[TaskArtifactStore.ts](../../src/services/planning/artifact/TaskArtifactStore.ts)

### 4.1 问题背景

SA 执行失败后 MB 重新派遣新 SA，新 SA 无法获取前序 SA 的中间成果（搜索结果、文件内容等），导致大量重复劳动。

### 4.2 自动提取策略

SA 工具只要返回结果，`SubAgentRunner.storeToolResultAsArtifact()` 就会自动提取并归档（成功和失败结果都保留），无需 LLM 显式指令：

| 工具 | 最大保留 | 数据类型 |
|------|---------|---------|
| `web_search` | 3000 字符 | `search_results` |
| `read` / `file_read` | 1500 字符 | `file_content` |
| `exec` | 500 字符 | `execution_output` |
| `file_write` | 200 字符 | `file_operation` |

### 4.3 双层注入架构

```
TaskArtifactStore（内存 30K token 总预算；写入超限时 FIFO 淘汰）
    │
    ├──── 索引注入（轻量）─────→ MasterBrainInputBuilder.build()
    │     工具名 + 来源参数               │
    │     预算：variableBudget × 10%     ▼
    │                              MB Prompt [TASK_ARTIFACTS]
    │                              帮助 MB 指引新 SA 利用已有成果
    │
    └──── 快照注入（完整数据）───→ SubAgentDispatcher.buildTaskContext()
          完整 Artifact 内容              │
          预算：总上下文窗口 × 15%        ▼
                                   SA System Prompt [前序任务成果]
                                   新 SA 获取搜索结果/文件内容
```

`TaskArtifactStore.getSnapshot(budgetTokens)` 的选择策略是：若完整内容放得下则全部注入；若预算不足，则从最新 Artifact 开始选择，优先保留最近写入的成果，同时索引仍包含全部轻量条目。

### 4.4 生命周期

```
AgentLoopFSMIntegration 构造       → 初始化 TaskArtifactStore
SubAgentRunner 工具返回结果         → storeToolResultAsArtifact() 写入
run() → reset()（每次用户请求开始） → 清空所有 Artifact
写入超过 30K token 总预算           → FIFO 淘汰最旧条目
```

## 五、Human-in-the-Loop (HITL) — 步间介入机制

**文件**：[hitlStore.ts](../../src/stores/hitlStore.ts) / [SubAgentRunner.ts 步间检查点]

### 5.1 介入流程

```
UI（HitlInterventionBar）
    │ pause(contextId)        ← 用户点击「⏸ 暂停」
    ▼
hitlStore（Zustand）
    │ pausedContexts.add(contextId)
    │
SubAgentRunner（步间检查点）
    │ while (!terminated) {
    │     checkAbortSignal()         ← 终止优先
    │     isPaused() → waitForResume() ← HITL 检查点
    │     callLLM()
    │     executeTools()
    │ }

用户输入介入消息 → resume(contextId, message)
    ├── additionalInstructions 注入  ← 当前 SA 下一步 LLM 调用可见
    ├── persistedIntervention 注入   ← 后续每步 LLM 尾部热区持续可见
    ├── messages[] 追加 user 消息     ← 当前 SA 生命周期内永久可见
    ├── TaskArtifactStore.write(user_intervention) ← 跨 SA 持久化
    └── emitObservation({ thinking: '🧑 [用户介入] ...' }) ← UI 时间线
```

### 5.2 竞争条件处理（preResolvedMap）

用户可能在 LLM 调用期间（`waitForResume` 尚未调用前）就点击恢复：

| 场景 | 处理方式 |
|------|---------|
| 正常：`waitForResume` 先等待，用户后恢复 | `resume()` 直接找到 resolver 调用 |
| 竞争：用户恢复在 `waitForResume` 调用之前 | 消息暂存 `preResolvedMap`，`waitForResume` 启动时立即消费 |

> [!IMPORTANT]
> `isPaused()` 必须同时检测 `pausedContexts` 和 Zustand state 中的 `preResolvedContexts`。`preResolvedMap` 是模块级缓存，不触发 React 重渲染，因此需要用 `preResolvedContexts` 让 UI 和 Runner 都能感知“已恢复但尚待消费”的状态。

### 5.3 介入消息持久化到 Task Artifact

介入消息以 `user_intervention` 类型写入 `TaskArtifactStore`：

```typescript
TaskArtifact {
    key: 'user_intervention_0',
    toolName: 'user_intervention',
    content: '[用户在第 N 步介入调整策略]\n用户指令: <原文>'
}
```

MB Prompt 的 `[TASK_ARTIFACTS]` Section 展示介入告警块，确保下一轮派遣的 SA 继承约束。新 SA 的 System Prompt 中，`hitlOverride` 被提升至 `buildInputProtocol()` 最顶部（CAUTION 告警块）；当前 SA 则通过 `persistedIntervention` 在每步 LLM 调用的尾部热区持续接收用户介入指令。

## 六、会话轮次隔离 — tool 消息生命周期

SA 的工具调用结果（如 web_search 的大量原文）可能导致父级 Session 膨胀，通过如下机制隔离：

| 机制 | 实现位置 | 作用 |
|------|---------|------|
| **SA 工具隔离** | `AgentLoop.createToolAdapter()` | `isSubAgentContext === true` 时跳过 `addMessage`，SA 内部工具结果不写入父级 Session |
| **轮次清理** | `AgentSession.clearToolMessages()` | 每轮 `runWithFSM()` 开始时清除上一轮的 `role=tool` 消息 |
| **SA observations 保留** | `SubAgentDispatcher.buildDispatchResult()` | SA 的最终汇报以 `role=tool` / `toolName=sub_agent_*` 写入父级 Session（当轮可用，下轮清除） |
| **MB 调用前转换** | `AgentLoop.callLLM()` | 纯文本 MB 调用不支持 `role=tool`，因此当轮 `tool` 消息会转换为带语义围栏的 `role=user` 消息 |

```
同一轮请求中:
  Session = [user₁...userₙ] + [assistant₁...assistantₙ] + [tool(当轮SA observations)]
  → MB 调用前将 tool 转成带 [END_SA_REPORT] 的 user 消息后可见

下一轮请求开始:
  clearToolMessages() → [tool] 全部清除
  Session = [user₁...userₙ] + [assistant₁...assistantₙ]
  → 跨轮知识传递由记忆系统（task_experience）承担
```

## 七、任务经验记忆 — SA 试错经验持久化

SA 执行中遇到有意义的试错经验时，在 `TASK_COMPLETE` 后附加 `## EXECUTION_EXPERIENCE` 标记，经验被提取后**同时注入 MB 和 SA 的 Prompt**：

```
SA 报告中包含 "## EXECUTION_EXPERIENCE"
    │
    ▼ ExperienceExtractor.extractExperienceFeedback()
    │
    ▼ pendingExperiences 累积（SharedState）
    │
    ▼ RESPOND_TO_USER 阶段批量 → MemoryService.saveTaskExperience()
    │                              (SQLite，task_experience 类别)
    │
    ├──── MB 端：MasterBrainPrompt [TASK_EXPERIENCE] Section
    └──── SA 端：SubAgentPromptBuilder 历史经验区块（buildTaskExperienceSection）
                 含去重提示，避免 SA 重复报告同类经验
```

写入条件：
- ✅ 环境配置/路径/编码踩坑
- ✅ 更高效的执行方式发现
- ✅ 工具/技能使用故障教训
- ❌ 一切顺利无需报告

## 八、安全增强机制汇总

| 机制 | 注入位置 | 作用 |
|------|---------|------|
| 执行边界负面清单 | `BASE_TEMPLATE`（始终） | 5 项明确禁止（修范围外 bug/安装包/推测操作等） |
| 工具调用 CoT 自检 | `TOOL_CALL_SELF_CHECK`（仅 Loop） | SA 每步声明操作意图，建立自我约束框架 |
| 尾部安全锚点 | `SAFETY_FOOTER_TEXT`（`subAgentSafetyFooterEnabled=true` 时，每步 LLM 调用） | 位于 context 尾部注意力热区，对抗长上下文稀释 |
| HITL 尾部强约束 | `persistedIntervention`（每步 LLM 调用，位于 Safety Footer 之后） | 用户介入后的指令持续拥有最高尾部注意力优先级 |
| 越权检测 Checkpoint | `MasterBrainPrompt.buildCheckpointEvaluationPrompt` | 检测 SA 操作资源是否超出委派范围 |
| 意图漂移检测 Checkpoint | 同上 | 检测 SA 下一步工具调用是否与任务目标无关 |
| ADJUST_STRATEGY 进度摘要 | `buildProgressSummaryForSA()` | 策略调整时注入已完成步骤，防止上下文截断后重复操作 |
| 语义围栏 | `AgentLoop.ts`（`sub_agent_*` tool 报告转 user 消息时） | 防止 MB 将 SA 报告误判为新用户请求 |
| MB 决策历史 | `MasterBrainPrompt` 的 `[MB_DECISION_HISTORY]` | 最近 5 轮 MB rationale/task/status，防止重复派遣与决策漂移 |
