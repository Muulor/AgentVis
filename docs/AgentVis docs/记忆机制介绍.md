# AgentVis 记忆机制介绍

> 文档范围: `src/services/memory/` 全模块 + `src/components/memory/` UI 层

---

## 一、设计目标

AgentVis 记忆系统的核心目标是：**让 AI Agent 跨会话、跨任务地"认识"用户**，并将有价值的认知沉淀为长期知识，而不是在每次对话中从零开始。

具体而言，记忆系统需要解决三个问题：

1. **当前会话的对话堆积**：随着对话轮次增加，LLM 上下文窗口被占满，如何在不丢失关键信息的前提下压缩？
2. **用户事实的跨会话持久化**：用户的身份、偏好、目标等稳定事实，如何可靠地识别并长期保存？
3. **上下文的精准供给**：保存的记忆如何在合适的时机、以合适的粒度注入到 LLM Prompt，而不引入噪音？

---

## 二、三层记忆架构

记忆系统采用三层分级存储，从"热"到"冷"依次为：

```
┌─────────────────────────────────────────────────────┐
│              Layer 1: 短期缓冲 (Short-Term Buffer)    │
│   - 存储最近 N 条 user 消息及其 assistant 回复          │
│   - 内存级（ShortTermBuffer）+ 数据库持久化双轨存储      │
│   - 水位线达阈值后向 Layer 2 压缩                       │
├─────────────────────────────────────────────────────┤
│              Layer 2: 摘要层 (Summary Layer)          │
│   - LLM 驱动的状态型摘要（含已确认决策、待决问题）        │
│   - 摘要向量化，支持 embedding + 临时 BM25/RRF 混合召回 │
│   - 通过 metadataJson 传递状态字段，供后续决策回溯        │
├─────────────────────────────────────────────────────┤
│              Layer 3: 长期事实 (Fact Layer)            │
│   - 结构化用户事实，按 6 大类别存储                      │
│   - 三阶流水线：候选扫描 → 稳定性验证 → LLM 提取写入     │
│   - importance 评分 + 类别汇总                           │
└─────────────────────────────────────────────────────┘
```

## 三、Layer 1：短期缓冲（ShortTermBuffer）

### 3.1 滑动窗口 + FIFO 淘汰

`ShortTermBuffer` 维护一个以 **user 消息数** 为基准的滑动窗口。

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `windowSize` | 10 | 最大保留的 user 消息数 |
| `watermarkThreshold` | 0.6 | 触发压缩的水位线（60% = 6 条） |
| `batchSizeRatio` | 0.4 | 每次弹出的批次比例（40% = 4 条） |

水位线基于 user 消息数而非总消息数，是为了避免 assistant 超长回复导致误触发。

### 3.2 双轨存储机制

每次用户发送消息并收到回复后，`MemoryService.addInteraction()` 会：

1. **内存写入**：调用 `buffer.addMessages()` 更新内存缓冲，供同一会话内快速访问
2. **数据库持久化**：通过 Tauri IPC 调用 Rust 后端 `memory_create` 命令，将 `User: <内容>` 和 `Agent: <内容>` 分别持久化至 SQLite

持久化采用**补偿策略**：写入 user 消息成功后，若写入 assistant 消息失败，则自动回滚已写入的 user 记录，避免产生孤立记录。

### 3.3 水位线转换流程

```
每次 addInteraction() → 检查数据库 short_term 记录数
│
├── 未达水位线（user 消息数 < windowSize × watermarkThreshold）
│   └── 不触发
│
└── 达到水位线
    ├── 发射 memory:watermark_triggered 事件（UI 显示"整理中"状态）
    ├── 从 FIFO 队列头部取 batchSize 条消息
    ├── 调用 SummaryManager 生成 LLM 摘要
    └── 摘要写入 Layer 2 后，删除对应 short_term 记录
        └── 循环检查，直至水位线降到安全区
```

水位线流程有**死循环防护**：如果连续两轮处理后 `short_term` 记录数未变化（LLM 持续失败），自动退出循环。水位线处理使用**互斥锁**（`_watermarkLock`），防止 `onSessionEnd` 与 `checkWatermarkOnResume` 并发调用导致重复摘要。

## 四、Layer 2：摘要层（SummaryManager）

### 4.1 状态型摘要设计

普通摘要只做内容压缩，AgentVis 的摘要是**状态型**的，包含以下字段：

| 字段 | 类型 | 含义 |
|------|------|------|
| `summary` | string | 高度概括的状态摘要 |
| `keyPoints` | string[] | 关键状态点（简洁条目） |
| `topics` | string[] | 讨论主题（用于语义检索） |
| `mentionedFiles` | string[] | 涉及文件路径 |
| `confirmedDecisions` | string[] | 已确认的结论/决策 |
| `openQuestions` | OpenQuestion[] | 待决问题（驱动精准原文回溯） |
| `invalidatedPoints` | string[] | 已失效的观点 |

这种设计的意图是：摘要不仅是压缩后的历史，更是**对话状态的快照**，能帮助 LLM 精准感知当前上下文的确定性与不确定性。

### 4.2 摘要的向量化与混合召回

每条摘要生成后会同步调用 Embedding API 建立向量索引（`MemoryVectorIndex`）。注入上下文时，`MemoryContextProvider` 调用 `MemorySummaryRetriever`：先用 embedding 召回语义候选池（默认最终 Top-3，相似度阈值 0.4），再临时构建 `BM25Index` 对摘要状态字段做词面召回，最后用 RRF 融合排序。

BM25 的索引文本来自 `content`、`topics`、`keyPoints`、`mentionedFiles`、`confirmedDecisions`、`openQuestions` 的问题/范围/关键词，以及 `invalidatedPoints`。BM25-only 候选必须命中文件名、路径、代码符号、引用短语、明确专名等强锚点才能保留；“之前、方案、问题、机制、记忆、摘要、这个、那个”等泛词不能单独触发召回。

如果 embedding 失败但 BM25 有强锚点命中，会直接返回这些强锚点结果；如果 embedding 与 BM25 都没有可用结果，则**降级**为返回最近 K 条摘要，并跳过"原文溯源"和"证据片段加载"，避免不相关内容污染 Prompt。

### 4.3 待决问题与证据回溯

当摘要中有 `openQuestions` 时，`EvidenceRetriever` 会：

1. 根据每个问题的 `keywords`、`turnHint` 和当前 `userQuery` 定位原始消息范围
2. 以用户轮次为单位成对提取 `[User] + [Assistant]` 证据片段，避免只召回用户问题或只召回助手回答造成语义断裂
3. 默认只给召回排序第一的摘要展开 1 轮证据；当 query 明确要求“完整过程、复盘、上下文、链路、timeline、recap”等历史脉络时，扩展到 2 轮
4. 长回答会按相关段落裁剪，再挂载到对应问题节点下的 `evidenceSlices`
5. 最终以 `精准回溯证据` 格式注入 Prompt，帮助 LLM 补全遗漏的细节

## 五、Layer 3：长期事实 — 三阶流水线

事实提取是整个记忆系统的核心，采用**三层渐进式流水线**，而非每次对话都调用 LLM 提取：

```
用户消息输入
     │
     ▼
【Layer 1 - 候选扫描】MemoryCandidateScanner（规则，不调用 LLM）
     │   扫描 user 消息，匹配意图词典
     │   输出: MemoryCandidate[]（候选列表，含分类和初始分数）
     ▼
【候选池合并】StabilityVerifier.mergeCandidatesAsync()
     │   新候选 ↔ 候选池语义匹配，同类相似项合并 occurrenceCount++
     ▼
【Layer 2 - 稳定性验证】StabilityVerifier（规则 + 语义增强，不调用 LLM）
     │   打分模型：见 5.2 节
     │   分数 ≥ 5 → 晋升到 Layer 3
     │   分数 3~5 → 保留在候选池等待下次累积
     │   分数 < 3 → 丢弃
     ▼
【Layer 3 - LLM 提取写入】FactExtractor.extractAndSaveFromVerified()
     │   调用 LLM 将候选精炼为标准化事实，写入长期记忆
     └── 发射 memory:facts_updated 事件（通知 UI 刷新）
```

### 5.1 事实类别（6 类）

| 类别 ID | 中文名 | 典型示例 |
|---------|-------|---------|
| `identity_role` | 身份/角色 | "我是后端工程师" |
| `preference_style` | 偏好/风格 | "我喜欢简洁的回复" |
| `long_term_goal` | 长期目标/约束 | "我在准备系统设计面试" |
| `knowledge_level` | 知识基线 | "熟悉 TypeScript，了解 Rust" |
| `interaction_signals` | 交互信号 | "使用中文交流，时区 UTC+8" |
| `task_experience` | 任务经验 | "Windows 下应用 findstr 而非 grep" |

`task_experience` 由 Sub-Agent 执行后自动调用 `MemoryService.saveTaskExperience()` 直写，**绕过三层验证流水线**，因为 SA 的执行结论具有天然的确定性。

### 5.2 稳定性打分模型

`StabilityVerifier` 对每个候选计算**稳定性得分**，决定候选的命运：

| 维度 | 分值 | 说明 |
|------|------|------|
| 跨多轮重复出现（`occurrenceCount ≥ 2`） | +3 | 同一事实在不同轮次出现 |
| 表达确定（无模糊词） | +2 | 无"可能/也许/估计"等词 |
| 影响决策路径 | +2 | 含"决定/必须/禁止"等词 |
| 被用户确认 | +3 | "对的/记住这个" |
| 含临时/情绪/假设词 | −3 | "烦/假如/暂时" |
| 强上下文绑定（工作/代码相关） | −2 | "这个项目/这段代码/刚才" |
| 类别加成（`identity_role/preference_style`） | +3 | 天然稳定的类别 |
| 类别加成（`long_term_goal/knowledge_level`） | +2 | 长期背景类较稳定 |
| 类别加成（`interaction_signals`） | +0 | 不在规则层加分，交由 LLM 判断价值 |
| 偏好软信号（仅 `preference_style`） | +1 | "喜欢/偏爱/讨厌" |

**阈值规则**：
- `score ≥ 5`：晋升，进入 LLM 提取
- `retentionThreshold ≤ score < 5`：保留在候选池，等待 `occurrenceCount` 累积（默认 `retentionThreshold = 3`）
- `score < retentionThreshold`：丢弃

`interaction_signals` 的保留阈值降为 1：这一类只在规则层过滤明显噪声，最终是否值得写入由 Layer 3 的 LLM 判断。

**灰度晋升（Cascading Verification）**：处于灰度地带（`retentionThreshold ~ 5` 分）的候选会额外调用语义嵌入分析（`SemanticAnchors`），对已有证据进行"放大"——若语义层确认确定性强，可将得分提升至晋升阈值。嵌入层**不能单独推送候选**（Gate-first），只能放大已有的规则信号。

### 5.3 候选池溢出晋升

若某一类别在候选池中积压候选数达到 5 条（`CANDIDATE_POOL_OVERFLOW_THRESHOLD`），无论单条分数是否达标，都将该类别的全部候选**批量晋升**到 LLM 提取阶段。这是对"缓慢积累型事实"的安全阀机制。

## 六、混合触发模型（MemoryTriggerManager）

触发模型决定"何时"启动三层流水线，采用**混合触发**策略，避免每轮对话都调用 LLM：

### 6.1 触发信号来源（多源）

| 信号类型 | 强度 | 触发阈值 |
|---------|------|---------|
| 显式记忆命令（"记住这个"） | 强信号 | +4.0 分，立即触发 |
| 用户确认词（"对的"/"就是这样"） | 强信号 | +3.0 分 |
| 候选扫描命中（每个候选 +0.5） | 弱信号 | 最多 +2.0 分/轮 |
| 语义分数累计达 5.0 | 语义触发 | – |
| 对话轮次达 10 轮 + 基本分数 ≥ 2.0 | 兜底触发 | – |
| 会话结束 / Agent 切换 / 任务完成 | 强制触发 | 有新内容即触发 |

### 6.2 生命周期强制触发

以下生命周期事件会调用 `triggerOnLifecycleEvent()`，执行**强一致点**检查：

- Agent 切换回来（`AgentChatView` 挂载时调用 `checkWatermarkOnResume()`）
- 用户关闭聊天窗口（`onSessionEnd()`）
- Task 模式任务完成后

生命周期触发通过比较 `latestMessageId` 与 `lastProcessedMessageId`，如有差异则触发，避免重复处理相同内容。

### 6.3 增量处理优化

每次触发后，系统记录 `lastProcessedMessageId`。下次触发时使用 `loadRecentMessagesAfter()` 增量加载，只处理上次处理点之后的新消息，避免全量重扫描。

## 七、上下文注入（MemoryContextProvider）

记忆的价值在于注入 Prompt。`MemoryContextProvider` 负责加载事实、召回摘要、补齐 Evidence Slices，并返回结构化记忆上下文；具体 Prompt 渲染由调用侧完成。

### 7.1 注入策略

| 数据 | 加载策略 | 注入方式 |
|------|---------|---------|
| 事实（`fact` 层） | **全量加载** | Chat 路径分为绑定事实/背景事实；MB 路径按 `factsByCategory` 分组渲染 |
| 摘要（`summary` 层） | **混合召回**（embedding 候选 + 临时 BM25/RRF，Top-3，阈值 0.4） | 时间线格式，按生成时间正序排列 |
| 原始消息（溯源） | 按需加载，受 `includeOriginal` 控制 | 附加到对应摘要后 |
| Evidence Slices | 按需加载（openQuestions 非空时） | 默认只展开首个相关摘要的 1 轮 User/Assistant 成对证据；历史脉络类 query 扩展到 2 轮 |

### 7.2 事实分区注入

事实全量加载后会按使用场景分区注入：

**Chat 路径绑定性事实**（模型必须遵守的约束）
```
# 已确认的身份与偏好

## 用户身份
- [identity_role 事实列表]

## 用户偏好
- [preference_style 事实列表]

> 注意：当前用户明确表达的意图优先于历史偏好
```

**Chat 路径参考性事实**（用户背景知识，不强制约束）
```
# 与用户交互产生的事实背景知识（仅供参考）

## 长期目标
## 知识背景
## 交互信号
```

**Master Brain 路径分组事实**（用于决策判断）
```
**Identity And Preferences:**
**Long-Term Goals:**
**Knowledge Background:**
**Interaction Signals:**
**Other Facts:**
```

**任务经验**（独立注入给 Master Brain，用于派发任务时避免 Sub-Agent 重复犯错）
```
# 历史任务执行经验
> 以下经验来自过去任务执行中 Sub-Agent 的试错总结...
```

### 7.3 摘要 Prompt 格式

```
## 早期对话状态

- [2026-04-06 10:30] 讨论了系统架构方案，决定采用 FSM 引擎...
  已确认决策:
    ✓ 采用 YAML 定义 FSM 状态机
  待决问题:
    ? 分布式锁如何实现 (architecture)
  精准回溯证据:
    [Turn 5 - User] "我们需要支持多实例部署..."
  已失效观点:
    ✗ 最初提出的数据库轮询方案
```

## 十、关键设计决策

### 10.1 为什么三层，不是两层？

- **Layer 1（短期缓冲）** 负责消化"对话流量"，用最低成本保存最近上下文
- **Layer 2（摘要）** 负责长期历史的"有损压缩"，保留状态意图
- **Layer 3（事实）** 负责"无损萃取"，提炼出稳定、跨会话可复用的用户属性

两层设计会导致"摘要既要压缩上下文，又要提炼事实"，目标冲突。

### 10.2 为什么候选池不直接送 LLM？

直接每轮调用 LLM 提取事实代价高昂，且大量噪音会混入长期记忆。候选池 + 稳定性验证是一个**代价-质量权衡**：

- 规则层（Layer 1/2）过滤掉 70%+ 的噪音，无需 LLM
- 只有真正稳定的候选才消耗 LLM token（Layer 3）

### 10.3 为什么摘要召回使用 Embedding + 轻量 BM25/RRF 而非全量？

随着使用时间增长，摘要数量会持续增长。全量注入会大量占用上下文窗口且引入不相关噪音。embedding 召回的核心价值是：**只注入与当前 query 相关的历史状态**，而不是所有历史。

在此基础上，`MemorySummaryRetriever` 使用一次性的 BM25/RRF 作为轻量纠偏层：文件名、路径、代码符号、引用短语、明确专名等强锚点可以补足 embedding 不稳定的场景；泛词命中的 BM25-only 候选会被 gate 丢弃，且融合后不会强行补满 `topK`。这样可以提升摘要召回精度，同时避免把知识库 RAG 的完整 `HybridRetriever` 链路引入记忆摘要层。

### 10.4 task_experience 的特殊地位

SA 执行任务后积累的"试错教训"（如"Windows 下用 findstr 而非 grep"）是系统层面的客观事实，而非用户主观陈述，因此：

- 直写 Layer 3，以 0.9 提取置信度映射到 `importance`，绕过候选验证
- 独立渲染为 `## EXECUTION_EXPERIENCE` 注入给 MB，而非混入用户事实
- 有相似内容自动去重合并（`saveFactV2` 内部语义相似度 ≥ 75% 视为同一条）
- 事实当前通过全量加载注入上下文，不在写入时主动创建事实向量索引

---

### 10.5 为什么长期事实不走向量召回？

长期事实（包括手动新增的用户背景事实）数量通常较少、内容精炼，且对 Agent 的用户理解具有稳定背景价值。当前实现选择**全量加载事实**，再由 Chat/MB 各自分区注入，而不是通过向量召回筛选事实。

这样可以避免用户手动维护的背景事实被检索阈值漏掉；事实合并仍由 `saveFactV2` 在同类别内做语义相似判断。删除事实时，UI 会调用带向量清理的后端接口，以兼容历史上可能存在的 `memory_fact_*` 索引。
